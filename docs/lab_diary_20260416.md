### 4. Root Cause Diagnosis: Kronos 23 kHz Cutoff

**The key discovery of the day.** We had already identified this in the prescan sessions, but today's systematic hardware benchmarks confirm it definitively:

**Kronos USB audio has a hardware analog anti-alias filter that hard-cuts at ~23 kHz.** The strongest plate resonances are at 26–95 kHz — completely invisible to Kronos. The census detects peaks using a 25 Hz swept sine (which can alias into the passband), but the actual on/off carrier separation at those frequencies is zero through the Kronos ADC path.

This explains everything:

- Hardware parity at chance → Kronos can't see the high-Q modes that carry the computation
- Simulation at 100% → the census frequencies are correct, the simulated feature vectors use the true mode amplitudes
- Polynomial features don't help → expanding noise gives more noise

## Root Cause Summary: Why Kronos Failed

| Factor              | Kronos path                  | PicoScope path                   |
| ------------------- | ---------------------------- | -------------------------------- |
| ADC resolution      | 24-bit                       | 8-bit                            |
| Analog bandwidth    | **23 kHz** (hardware filter) | **100 MHz**                      |
| Plate modes visible | Few (below 23 kHz)           | All (26–95 kHz)                  |
| Mode yield          | 44–57 peaks                  | 160–191 peaks                    |
| On/off separation   | 0 strong modes (prescan)     | Good separation (reservoir demo) |
| Parity accuracy     | 50% (chance)                 | **100%**                         |

**The Kronos has 16× better resolution but can't see the signals.** 24-bit precision on zero signal is still zero. The PicoScope's 8-bit ADC with 100 MHz bandwidth sees every plate mode and captures enough dynamic range for 100% classification.

The ideal sensor front-end needs both: **16+ bit resolution AND >100 kHz bandwidth**. The MCC USB-1608G ($469, 16-bit, 250 kS/s) would provide this.

---

## What We Learned

### A. Bandwidth Dominates Resolution for Reservoir Computing

The Kronos (24-bit, 23 kHz BW) fails completely while the PicoScope (8-bit, 100 MHz BW) succeeds perfectly. For spectral reservoir computing, seeing ALL the modes matters more than measuring each one precisely. This has direct implications for sensor front-end design: choose bandwidth over dynamic range.

### B. Polynomial Feature Expansion Is Essential

Raw linear features (mode amplitudes alone) give 37–62% parity — near chance. Polynomial expansion (degree ≤ 4) of the spectral features pushes to 100%. The glass physics provides nonlinear mode interactions, but extracting them requires reconstructing cross-mode products in the feature space. This is a software operation on the readout ASIC, not additional hardware.

### C. The Plate Outperforms Software Neural Networks

At every bit width (3–6 confirmed, 7 running), the plate achieves 100% parity while the best software baseline (MLP with h=64 hidden units) degrades from 75% → 57.8%. Both classifiers receive the same feature vectors. The plate's physics provides a computational advantage that software can't replicate from the same inputs.

### D. Hardware Parity Gaps Reveal Capture Path Problems

The stark gap between simulation (100%) and hardware (50%) on Kronos was the diagnostic clue. When simulation works and hardware doesn't, the bug is in the capture pipeline, not the physics. Prescan with on/off carrier separation measurements confirmed this quantitatively.

### E. Hybrid Path Has Promise But Needs Engineering

PicoScope AWG → Kronos ADC should combine clean DDS with 24-bit depth, but USB resource contention and the Kronos analog cutoff limit practical results. The better path is PicoScope AWG → dedicated wideband ADC (MCC USB-1608G).
